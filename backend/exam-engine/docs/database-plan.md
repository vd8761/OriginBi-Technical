# OriginBI Exam Engine — Database Plan (v2, aligned with Exam Portal Plan)

This rewrites the schema to match the Exam Portal Plan you authored:
4 audiences (Admin, Organization Admin, Organization User, Individual User), every assessment capability is a plugin, strict top-down entitlement cascade (Admin → Org → Exam → Question), and frozen exam snapshots so config changes never disturb live attempts.

The `users` table is **assumed to exist** with at least `id`. Nothing here touches it.

Stack split (per your architecture):
- **Next.js** — Admin / Org Admin / Org User / Individual web UIs
- **NestJS** — identity, RBAC, tenant management, exam authoring, scheduling, monitoring APIs, reporting
- **Go (this engine)** — exam session runtime, server-authoritative timer, autosave, scoring execution, Judge0 pipeline, telemetry ingest

Both NestJS and Go talk to the same Postgres. This schema is the contract between them.

## Design principles

1. **Snapshot, don't reference live content.** When an exam goes live, every reference is frozen to a `_versions` row. Editing source content never mutates an in-progress or completed attempt.
2. **Polymorphic plugin payloads.** Each question/answer carries a plugin discriminator and a `payload jsonb`. Plugins register a JSON Schema for their shape; the engine treats the body opaquely.
3. **Top-down entitlement cascade is data, not code.** Four separate entitlement tables (platform default → org → exam → exam-question) with the same `(state, config)` shape. The engine resolves the effective state by walking down. This is how plugins like *Coding* can be enabled at org level, restricted at exam level (Java only), and further restricted per question.
4. **Append-only telemetry.** `attempt_events` and `attempt_heartbeats` are partitioned and never updated. Aggregates live in `attempt_event_summary`.
5. **Owner = organization.** Individuals get an auto-created personal org (`kind = 'individual'`); platform-owned content lives in a `kind = 'system'` org. One ownership model serves all 3 audiences.
6. **All ids are UUIDv7** generated by the Go service for index locality, except telemetry which uses `BIGSERIAL`.
7. **TIMESTAMPTZ everywhere.** Soft-delete via `deleted_at` on user-authored tables; telemetry is dropped wholesale by partition rotation.

```sql
-- Plugin entitlement (your doc: Disabled, Enabled, Restricted)
CREATE TYPE plugin_state AS ENUM ('disabled', 'enabled', 'restricted');

-- Org kind
CREATE TYPE org_kind AS ENUM ('individual', 'corporate', 'college', 'system');

-- Exam template/exam lifecycle (your doc: Draft, Published, Scheduled, Live, Paused, Completed, Archived)
CREATE TYPE exam_status AS ENUM (
    'draft',        -- editable, not yet released
    'scheduled',    -- published with a future window
    'published',    -- ready to take (within window if scheduled)
    'live',         -- has any attempt currently in_progress
    'paused',       -- admin paused (new attempts blocked, existing frozen)
    'completed',    -- window closed AND no live attempts
    'archived'      -- retired, kept for audit
);
-- live / paused / completed are derived facts but stored to avoid recomputation.
-- Triggers / Go service maintain them. draft|scheduled|published|archived are author-set.

-- Attempt lifecycle (your doc: Assigned, Started, In Progress, Submitted,
-- Timed Out, Under Review, Evaluated, Published, Cancelled)
CREATE TYPE attempt_status AS ENUM (
    'assigned',         -- assignment exists, candidate hasn't opened
    'started',          -- candidate opened the exam, hasn't progressed past instructions
    'in_progress',      -- actively answering
    'paused',           -- internal: temporary disconnect under grace window
    'submitted',        -- candidate clicked submit
    'timed_out',        -- timer expired and engine auto-submitted
    'under_review',     -- submitted, evaluator queue picked it up
    'evaluated',        -- all evaluations completed; score finalized
    'published',        -- result released to candidate
    'cancelled'         -- admin cancelled (rule violation, technical issue, etc.)
);

-- Coding evaluation states (your doc: Queued, Running, Auto Evaluated,
-- Pending Manual Review, Manually Reviewed, Published)
CREATE TYPE evaluation_status AS ENUM (
    'queued',
    'running',
    'auto_evaluated',
    'pending_manual_review',
    'manually_reviewed',
    'failed',                 -- worker error; will retry
    'superseded',             -- a later evaluation replaced this one
    'published'
);

CREATE TYPE evaluator_kind AS ENUM ('auto', 'manual', 'llm');

-- Assignment state (your doc: assignments produce attempts that go Assigned → ...)
CREATE TYPE assignment_status AS ENUM ('pending', 'active', 'expired', 'revoked');
```

### Entitlement tables — one per scope, identical shape

Four tables, all FK'd to their scope. The engine walks them top-down to resolve the effective state for a (plugin, target) pair.

```sql
-- Level 1: platform default — ignored if you trust plugins.enabled_by_default;
-- this table exists for explicit overrides (e.g. globally disable a buggy plugin).
CREATE TABLE platform_plugin_entitlements (
    plugin_id       UUID PRIMARY KEY REFERENCES plugins(id) ON DELETE CASCADE,
    state           plugin_state NOT NULL,
    config          JSONB NOT NULL DEFAULT '{}',
    updated_by      BIGINT REFERENCES users(id),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- Level 2: per-organization (Admin grants / restricts capability to a corporate / college)
CREATE TABLE org_plugin_entitlements (
    id              UUID PRIMARY KEY,
    org_id          UUID NOT NULL REFERENCES organizations(id) ON DELETE CASCADE,
    plugin_id       UUID NOT NULL REFERENCES plugins(id) ON DELETE CASCADE,
    state           plugin_state NOT NULL,                 -- disabled | enabled | restricted
    config          JSONB NOT NULL DEFAULT '{}',           -- e.g. {"allowed_languages": ["python","java"]}
    granted_by      BIGINT REFERENCES users(id),
    granted_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (org_id, plugin_id)
);
CREATE INDEX ope_org_idx ON org_plugin_entitlements(org_id);

-- Level 3: per-exam (Org Admin disables a plugin, restricts languages, etc., within Admin's allowance)
CREATE TABLE exam_plugin_entitlements (
    id              UUID PRIMARY KEY,
    exam_version_id UUID NOT NULL REFERENCES exam_versions(id) ON DELETE CASCADE,
    plugin_id       UUID NOT NULL REFERENCES plugins(id) ON DELETE CASCADE,
    state           plugin_state NOT NULL,
    config          JSONB NOT NULL DEFAULT '{}',
    UNIQUE (exam_version_id, plugin_id)
);

-- Level 4: per-exam-question (e.g. this specific coding question only allows Python)
CREATE TABLE exam_question_plugin_entitlements (
    id                  UUID PRIMARY KEY,
    exam_question_id    UUID NOT NULL REFERENCES exam_questions(id) ON DELETE CASCADE,
    plugin_id           UUID NOT NULL REFERENCES plugins(id) ON DELETE CASCADE,
    state               plugin_state NOT NULL,
    config              JSONB NOT NULL DEFAULT '{}',
    UNIQUE (exam_question_id, plugin_id)
);
```

#### Resolution rules (implemented in a SQL function or Go helper)

```
effective(plugin, exam_question_id) =
    1. start with plugins.enabled_by_default
    2. apply platform_plugin_entitlements.state if present
    3. apply org_plugin_entitlements.state if present (org of the exam owner)
    4. apply exam_plugin_entitlements.state if present
    5. apply exam_question_plugin_entitlements.state if present
    => return the most-specific (state, merged config)
```

Lower layers can only further restrict; they cannot re-enable something the layer above disabled. The Go service enforces this on writes (e.g. an exam-level entitlement INSERT is rejected if the org-level state is `disabled`).

#### Coding-specific defaults (per your doc)

When `code.judge0` is enabled for an org, seed:
- `auto.testcases` → state=`enabled`
- `manual.review` → state=`enabled`
- `llm.openai` → not seeded; remains disabled unless Admin sets it explicitly

Allowed-language config lives in the entitlement `config` blob:
```json
{ "allowed_languages": ["python", "javascript", "java", "cpp", "c"] }
```
Default at org level if Admin doesn't set it: all 5 enabled. Org Admin can narrow at exam or question level.

## 4. Exam Templates vs Exams

Templates are admin-published reusable structures (your doc: *"Admin publishes reusable templates for organizations"*). Orgs clone a template into their own `exams` row, then customize within their plugin entitlements.

```sql
CREATE TABLE exam_templates (
    id                  UUID PRIMARY KEY,
    org_id              UUID NOT NULL REFERENCES organizations(id),  -- usually the system org
    name                TEXT NOT NULL,
    description         TEXT,
    audience            TEXT NOT NULL,    -- 'individual' | 'organization' | 'campus_hiring' | 'screening' | 'practice'

CREATE TABLE exam_template_versions (
    id                  UUID PRIMARY KEY,
    template_id         UUID NOT NULL REFERENCES exam_templates(id) ON DELETE CASCADE,

```

When an Org Admin creates an exam from a template, the engine copies the structure into `exams` + `exam_versions` (then re-validates against the org's entitlements).

#### Frozen snapshot

When `exam_versions.status` transitions to `published`, the Go service writes the full structure (sections + question payloads + entitlements + proctoring config) into `exam_versions.snapshot jsonb`. Every attempt loads this snapshot, never the live tables. Mid-exam author edits affect future attempts only.

CREATE INDEX attempt_events_attempt_idx ON attempt_events (attempt_id, occurred_at DESC);
CREATE INDEX attempt_events_kind_idx    ON attempt_events (kind, occurred_at);
CREATE INDEX attempt_events_severity_idx ON attempt_events (severity, occurred_at) WHERE severity >= 2;
```

## Hot-path query patterns

| Use case | Query | Indexes used |
|---|---|---|
| Resume an in-progress attempt | `SELECT … FROM attempts WHERE candidate_user_id = ? AND status IN ('started','in_progress','paused')` | partial index `attempts_status_live_idx` |
| Org Admin: pending reviews | `SELECT … FROM evaluations WHERE status = 'pending_manual_review'` join `attempts` filtered by org | `evaluations_queue_idx` |
| Reviewer: my open queue | `SELECT … FROM manual_review_assignments WHERE reviewer_user_id = ? AND completed_at IS NULL` | partial `mra_reviewer_open_idx` |
| Live monitor: violations on an attempt | `SELECT * FROM attempt_events WHERE attempt_id = ? AND severity >= 2 ORDER BY occurred_at DESC` | `attempt_events_severity_idx` + partition prune |
| Live monitor: counts per attempt | `SELECT kind, count, last_at FROM attempt_event_summary WHERE attempt_id = ?` | PK |
| Most recent code run | `SELECT … FROM code_runs WHERE answer_id = ? ORDER BY started_at DESC LIMIT 1` | `code_runs_answer_idx` |
| Connectivity timeline | `SELECT * FROM attempt_connectivity_gaps WHERE attempt_id = ?` | `gaps_attempt_idx` |
| Effective plugin state for a question at runtime | join the 4 entitlement tables (cached in Go process) | PKs |

```

                                                          │
                                  any in_progress attempt │
                                                          ▼
                                                        live
                                                          │ window closes & no live attempts
                                                          ▼
                                                      completed ─archive─▶ archived

assignment created ──▶ assigned ──open──▶ started ──answer──▶ in_progress

                                                  disconnect ▶ paused ◀ heartbeat resumes
                                                                 │
                                                       submit / timer / kill
                                                                 │
                                                          submitted | timed_out | cancelled
                                                                 │
                                                                 ▼

                                                                 │
                                                                 ▼

queued ──▶ running ──▶ auto_evaluated ──manual review required?──▶ pending_manual_review

## Plugin extension cookbook (how to add a new capability)

| New plugin | Tables touched | Steps |
|---|---|---|
| New question type (e.g. *SQL Playground*) | `plugins`, `question_versions.body`, optional sidecar | Register `sql.playground` plugin with body schema; engine treats body opaquely |
| New proctoring signal (e.g. *clipboard hash*) | `plugins`, `attempt_events.kind` | Frontend emits events with new `kind` slug; severity per config |
| New evaluator (e.g. *plagiarism check*) | `plugins`, `evaluations.plugin_id` | Worker subscribes to `evaluations` filtered by `plugin_id` and `status='queued'` |
| New feature flag (e.g. *per-question timer*) | `plugins`, entitlement tables | Engine checks effective state at runtime |
| Org-level config (e.g. LLM model + key) | `org_plugin_entitlements.config` | Read at evaluator start |

## Open decisions for you (carried + new)

1. **UUID strategy** — UUIDv7 from Go (recommended for index locality on hot tables) vs Postgres `gen_random_uuid()` v4.
2. **Telemetry retention** — drives partition rotation. 90 days? 1 year? Org-configurable?
3. **Result snapshot vs live recompute** — `result_publications.snapshot` is a frozen JSON of what the candidate sees; alternative is to render from live tables every request. Recommend frozen for audit.
4. **Reviewer queueing** — round-robin via app code, or pgmq / Postgres `LISTEN/NOTIFY`?
5. **Where audit log lives** — `audit_log(actor_user_id, target_kind, target_id, action, diff jsonb, occurred_at)` for authoring changes (who edited which question, who toggled an entitlement). Recommend: yes, single table, partitioned monthly.
6. **NestJS ↔ Go boundary** — does NestJS write to Postgres directly, or always call Go for runtime tables (`attempts`, `attempt_events`, `code_runs`, etc.)? Recommend the latter to keep timer/scoring authority in one process.
7. **Multi-tenant isolation** — row-level via `org_id` (current plan, simpler) vs Postgres RLS policies (stronger guarantee, more ops). Recommend row-level for v1.
